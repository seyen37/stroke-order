# 漢字基因 5000 會意字初稿 — 參考資料分析

> 分析對象：`5000.zip` (5000.TXT / 5000.htm / 29 個 BMP 檔)
> 發現：**這是整個參考系列的中間層 — 字符組合結構資料庫**

---

## 一、文件性質

`5000.TXT` (UTF-16 LE, 435 KB) 是 2012 年《字易》一書的**節錄版本**，來源是「漢字基因」計畫的初稿。依照命名、字庫格式、BMP 使用倉頡碼作為檔名的慣例判斷，這份資料與前兩批 SCG / MODVEC 字庫屬於**同一作者體系**（朱邦復先生的字根計畫）。

共有兩大部分：

| 段落 | 內容 | 規模 |
|---|---|---|
| 一～六（line 1–650） | 理論說明 + **509 個核心字**（242 象形 + 38 指事 + 267 會意） | 作為「漢字基因」的骨幹 |
| 七、附錄（line 651–end） | **5000 個會意字組合資料** | 3,756 entries / 3,645 unique characters |

加上 29 個 18×18 BMP 檔，用來表示沒有 Unicode 對應的**古字/衍字/倉頡組合形**（檔名 = 倉頡碼，如 `ANWG.BMP`、`RXDDO.BMP`）。文中以 `&~XXXX;` 占位符引用之。

## 二、資料結構（關鍵）

每一個會意字的 entry 有固定的六欄格式：

```
【字】  首[字首根] (體/用)-字首定義    尾[字尾根] (體/用)-字尾定義    類別-最早字形  概念定義
```

具體範例：

```
【明】  首[日] 體-太陽       尾[月] 體-月亮       會意-甲骨文   日月皆有光，可見可知，明顯狀。
【溫】  首[水] 體-液體       尾[昷] 用-適宜之影響  會意-小篆    水適宜之影響，冷熱感最宜人程度。
【閶】  首[門] 體-出入通道    尾[昌] 用-盛壯       會意-小篆    高大盛壯的門。
```

**資料欄位涵義**：

| 欄 | 語意 | 對我們的價值 |
|---|---|---|
| 字 | 目標字 | Primary key |
| 首/尾 | 字根 (head/tail) | **組件樹的左右分支** |
| 體/用 | 形義角色 | 體 = 形狀來源 (取形)，用 = 義涵來源 (取義) |
| 類別 | 會意/象形/指事/衍文 | 分類標籤 |
| 字形最早 | 甲骨/金文/小篆/隸書/楷書/衍文/石文 | 歷史演變版本 |
| 概念定義 | 該字的語義 | 可作教學/搜尋 |

**這是一個乾淨的 key-value 資料庫**：把每個字分解為兩個更原子的字根，直到觸底（象形或指事）為止。遞迴下去就是完整的字符組合樹。

## 三、與前兩批資料的整合關係

這份 TXT 正好填補了先前架構缺的那一環：

```
┌─────────────────────────────────────────────────────────────┐
│ L3  會意字 compounds     ← 【本 5000.TXT 附錄】3,645 個         │
│                              head + tail 遞迴分解             │
└──────────────────┬──────────────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────────────┐
│ L2  象形/指事 roots       ← 【本 5000.TXT 第一部分】 242 + 38   │
│                              含字形演變 (甲骨→金文→小篆→...)    │
└──────────────────┬──────────────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────────────┐
│ L1  筆畫 primitives       ← 【MODVEC.ASM 2010】 9 種參數化      │
│                              DTWC/JBMM/WFR/CPI/QAMO/QKMF/...  │
└──────────────────┬──────────────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────────────┐
│ L0  向量輸出 (vector out)  ← 【我們要新寫的部分】                 │
│                              polyline → SVG / G-code / JSON  │
└─────────────────────────────────────────────────────────────┘
```

**三批資料是同一個體系**：

- 1992 SCG：bitmap 實作版
- 2010 fontdata：vector 參數化實作版（還是出 bitmap，但內部是 vector）
- 2012 5000.TXT：**邏輯 schema** — 把字元如何拆解寫成人類可讀

## 四、樣本數驗證

| 項目 | 數量 | 說明 |
|---|---|---|
| 文件開頭宣稱 | 「約 500 (實得 509) 最基本字形」 | 基礎字根 |
| 象形 + 指事 entries | 262 + 38 = ~280 | L2 原子層 |
| 會意 entries（含正文+附錄） | 302 + 3,756 = **4,058** | L3 組合層 |
| 附錄獨立字 | **3,645 unique chars** | 去重後 |
| 聲稱涵蓋 GB 2312-80 | 90% of 6763 字 | 約 6,087 字可以用這套組合原理推出 |

用 3,645 個會意字 + 280 個字根 = **~3,925 個 key**，可以覆蓋 6,000+ 字的分解 —— 因為會意字本身又可作為更高階字的字首/字尾（遞迴複用）。這正符合先前 SCG 中 3 層 NBUFH 的嵌套結構。

## 五、BMP 附屬檔的用途

29 個 BMP 檔用於表示**電腦沒有的字形**（衍文、古字、組合字根）：

| 檔名（倉頡碼） | 出現位置 | 推測用途 |
|---|---|---|
| `HH1.BMP` | `象&~HH1;形` | 反形：手掌反面 |
| `HFDK.BMP` | 「奧」的尾部 | 雙手持物 |
| `WOBT2.BMP` | 「昷」原字 | 囚字的古寫 |
| `RXDDO.BMP` | 「理論基&~RXDDO;」 | 某特殊字 |
| `HH1.BMP`, `HM2.BMP`, 等 | 各種組合 | 未入 Unicode 的字根 |

這些檔案都是 18×18 1-bit 點陣。**對我們而言**：
- 可以直接向量化（18×18 用邊緣追蹤即可轉 SVG path）
- 或在 IR 裡標記為「無 Unicode」的 special token，用檔名（倉頡碼）作為 stable ID

## 六、對專案的實際意義

### 新增一個核心 IR 元件：`Decomposition`

先前我們規劃的 Component 還是籠統的「tree」。現在可以定義精準的欄位：

```python
@dataclass
class Decomposition:
    char: str                       # 目標字，例如「明」
    head: ComponentRef              # 字首
    tail: ComponentRef              # 字尾
    category: Literal['象形', '指事', '會意', '衍文', '異體']
    earliest_form: str              # '甲骨' / '金文' / '小篆' / '隸書' / '楷書' / '衍文' / '石文'
    concept: str                    # 概念定義 (供搜尋/教學)

@dataclass
class ComponentRef:
    root: str                       # 字根字元 例如「日」
    role: Literal['體', '用']        # 取形 / 取義
    definition: str                 # 字根在該字中的具體定義
    # 由於可能是沒有 Unicode 的字，root 可為倉頡碼占位符例如 "@ANWG"
```

### 簡化 IR stack

原本規劃的「Character → ComponentTree → Stroke」可以具體化為：

```
解析 5000.TXT → Decomposition database (in-memory dict)
                │
                ▼
        查詢「明」→ head=日 + tail=月
                │  遞迴直到觸及 atom (象形/指事)
                ▼
        atom 字根 → 連到 Stroke primitives (MODVEC 9 種)
                │
                ▼
        Stroke + FontStyle → polyline → exporter
```

### 具體可直接用

1. **5000.TXT 解析器**：純正規表達式，一個下午可寫完。Output 直接填入 Decomposition dict。
2. **字首/字尾位置規則**：文中有明確定義 —— 「字首位於字形上方、外圍或左側，字尾則居其餘空間」。可以硬編成 layout rule。
3. **體/用 分類**：渲染時**只依 `體` 的字根取形**，`用` 類只影響釋義，不影響筆跡。這大幅簡化向量生成邏輯。
4. **歷史字形作為 style parameter**：MODTRACE 有 `BBFNTKND` (明/黑/圓)，這裡多出一個維度：甲骨/金文/小篆/隸書/楷書。五套字形層疊，理論上可以做「時光機」式的字形演化動畫（文件原意就是教學動畫）。

## 七、更新後的路線圖

前一份分析 (`REF_ANALYSIS_SCG2010.md`) 建議走「路線 C 混合」：MMH 起步 + 挖 MODVEC 筆畫原子。

**加入 5000.TXT 後，現在可以走「路線 D：直接用你手上的完整資料」**。

| Phase | 原計畫 (路線 C) | 更新計畫 (路線 D) |
|---|---|---|
| Phase 1 | MMH 載入 + IR 骨架 | **5000.TXT 解析**，Decomposition DB，約 3,645 字覆蓋率 |
| Phase 2 | 三種 Exporter | 同上，但 Stroke 從 MMH medians 借用 |
| Phase 3 | 本地 Web 介面 | 同上 |
| Phase 4 | 筆畫範本庫 | **反解 MODVEC 筆畫原子**，得 9+ 參數化 primitive |
| Phase 5 | TTF 整合 | **五套歷史字形風格**（甲骨→楷書）+ 三套現代字型（明/黑/圓） |

**路線 D 的優勢**：不用 MMH，整條鏈都是你手上的繁體資料，且符合你描述的「從筆順範例拆解出共用的向量筆跡」初衷 —— 這正是 5000.TXT 資料庫的本質。

**代價**：
- 沒有 MMH 那種現成的 polyline medians，L1→L2 這層要自己補
- 可先把 5000.TXT 的字根（280 個象形/指事）用 **Unicode 字型 + FontForge outline → skeletonize** 產出 polyline，作為每個字根的「標準筆跡」。之後再做參數化。

## 八、結論

1. **5000.TXT 是整個系列的 schema 文件** —— 前兩批的 bytecode 現在有了對照表，每個字為什麼這樣拆、為什麼這樣組，都寫得清清楚楚。
2. **3,645 個會意字的 head/tail 分解**是立刻可用的結構化資料，解析成本極低（兩個小時內可搞定）。
3. **建議走路線 D**：完全用手邊的繁體資料建立 L0-L3 四層系統，路線 C 的 MMH 可以備用但不是主線。
4. **Phase 1 的第一個 commit 應該是「5000.TXT 解析器 + Decomposition DB」**，而不是先處理筆畫。有了結構骨架，後面填筆畫資料才有歸屬。
