# 2026-05-01 — Phase 11 印章模組精修

> 印章模組四個關鍵決策：1+2 layout 採非均勻拉長、outline 字型改 bbox-center 對齊、scale 從 EM-based 改 bbox-based、篆書鋸齒以 `shape-rendering` 緩解而非換字型。

## 背景

Phase 9-10 完成字型授權 gate（CC BY-ND 條款的「主動同意」UX）後，使用者試用印章模組時陸續回報問題。Phase 11 一日內處理完 8 個子問題（11a-11h），其中 4 個牽涉設計決策，記錄如下。

## 決策 1：3 字姓名章採台灣傳統 1+2 layout（非均勻拉長）

### 選項

| 選項 | 描述 | 視覺 |
|---|---|---|
| A | 3 字垂直排單欄（rows=3, cols=1） | 一條長直線 |
| B | 3 字横一排（rows=1, cols=3） | 三個正方形 |
| C | 1+2 layout：右姓拉高、左 2 字縮小堆疊 | 傳統台灣印章 |

### 選 C

**為什麼**：用戶提供台灣傳統印章圖例，C 是業界標準。A、B 視覺上不傳統，且 3 字本來就是難題（無法 2×2 整齊排）。

**代價**：placement 結構從 4-tuple `(c, x, y, rot)` 改成 6-tuple `(c, x, y, rot, w_mm, h_mm)`——支援 per-char 非均勻 size。新 helper `_char_cut_paths_stretched` 分離 `scale_x` / `scale_y`。所有 preset 的舊 4-tuple 路徑都要 migrate。

**參數**：右 surname `width=0.5×inner_w, height=0.92×inner_h`（≈1.84:1 拉長比），左 2 字各 `min(inner_h×0.46, inner_w×0.50)`。

### 教訓

- **業界視覺標準有時不能被工程結構限制**。原 4-tuple 結構假設「所有 cell 大小一樣」，這個假設在 3 字的情境下被破。
- **重構 placement 結構的時機**：當「現有 grid scheme 撐不住一個 layout 變體」時就重構，不要疊 hack。

## 決策 2：outline 字型改 bbox-center 對齊（不用 EM-center）

### 問題

用戶回報：印章切到隸書（國家發展委員會典型 TTF）字心偏下。視覺上感覺整個字往下掉。

### Root cause

TTF / OTF 字型的 typographic baseline 不在 EM 中心（座標 1024/2048），而在 ascender 附近（座標 ~1554/2048）。EM-center 對齊讓 baseline 落在 cell 上方 ~25% 處，視覺上字身偏下。

### 選項

| 選項 | 描述 | 缺點 |
|---|---|---|
| A | 用 baseline-aware shift（讀字型 metric） | 需要解 OTF metric tables，複雜 |
| B | 用 bbox-center 對齊（取 outline path 的真實 bbox） | 每字算一次 bbox |
| C | 維持 EM-center + 加固定 y offset | 不同字型偏移量不同，magic number |

### 選 B

**為什麼**：B 對所有 outline 字型都適用，不需要解字型內部 metadata，就靠已 parse 完的 outline path bbox。每字 bbox 算一次成本可忽略（OTF 字 outline 通常 < 500 個 cmd）。

**代價**：新 helper `_char_outline_bbox_full_em(c)` / `_char_outline_bbox_em(c)`，patch.py 與 stamp.py 都要切過去用。

### 教訓

- **跨資料源的座標系統不能假設統一**。本 repo 同時混用 stroke-bearing fonts (KanjiVG / G0V) 與 outline-only fonts (CNS / 隸書 / 宋體 / 篆書)，rendering pipeline 對兩類來源的座標處理必須分離。
- 這類 multi-source rendering 的「異質性債務」要早期就有設計層面的隔離 (per-source coordinate normalizer)，不要等到視覺問題出現才補。

## 決策 3：scale 從 EM-based 改 bbox-based（字撐滿 cell）

### 問題

決策 2 解了字心位置，但用戶接著回報「字還是太小，沒撐滿格子」。

### Root cause

原本的 EM-based scale `size_mm / 2048` 假設「整個 EM 都是字身」。實際上 outline bbox 通常只占 EM 的 70-80%（字身周圍有 padding），所以視覺上字明顯比 cell 小一圈。

### 選項

| 選項 | 描述 |
|---|---|
| A | 維持 EM-based + 用戶手動調 size 參數 | UX 差 |
| B | bbox-based scale：`scale_x = w_mm / bbox_w_em` | 自動 |
| C | 加 padding ratio 參數讓用戶控制 | 增加 API 表面 |

### 選 B

**為什麼**：用戶要的是「填滿格子」，B 直接做這件事。bbox-based scale 在決策 2 已經算過 bbox 了，B 的成本是「重用」現有資料。

**代價**：失去 EM-coherent scale（同 EM 不同 outline 的字會被各自縮放成同 bbox 大小），但對印章 use-case 是 feature，不是 bug——印章本來就要每字撐滿格子。

### 教訓

- **「正確的 scale」取決於 use-case**。書寫文件用 EM-coherent scale（字大小有自然比例），印章用 bbox-coherent scale（字撐滿格子）。同一個 codebase 兩種需求並存時要分開處理。

## 決策 4：篆書邊緣鋸齒以 `shape-rendering` 緩解（不換字型 / 不加 SVG filter）

### 問題

用戶：「篆體字，字體邊緣不平滑，請問可再平滑些嗎？」

### Root cause（curl 直接看 SVG 出來）

崇羲篆體 OTF 內單字 outline path 的命令分布：
- **142 段 `L` (直線)**
- **181 段 `Q` (二次 Bezier)**
- **0 段 `C` (三次 Bezier)**

→ 字型作者用大量短直線近似平滑曲線。這是 OTF 設計選擇，不是 stroke-order 渲染 bug。bbox-based scale (決策 3) 放大後直線片段間的視覺鋸齒就明顯。

### 選項

| 選項 | 描述 | 成本 | 效果 |
|---|---|---|---|
| A | 換另一套 TrueType 篆體字型（用 C 命令的） | 高（找字型 + 授權處理 + LICENSE 補 attribution + 加進 fetch_fonts.sh） | 治本 |
| B | 自訂 spline interpolator 把 L 序列重 fit 成 C | 極高 | 治本但工程量大 |
| C | SVG 加 `shape-rendering="geometricPrecision"`，瀏覽器精度優先 anti-aliasing | 極低（2 行 attribute） | 緩解 |
| D | SVG 加 `<filter>` Gaussian blur 後處理 | 低 | 邊緣模糊，破壞字型風格 |

### 選 C

**為什麼**：在 effort/value ratio 上，C 完勝。A、B 是治本但成本對應價值不高（篆體只是 5 套字型之一，且印章模式才會用到）。D 雖然簡單但會破壞字型風格。

**代價**：邊緣鋸齒不會完全消失，只是 anti-aliasing 品質提到瀏覽器最高檔。如果未來用戶再次要求更平滑，再升級到 A。

**已寫**：commit message 與 work log 都註明「天花板下的 best effort」，未來若有需要可升級到 A。

### 教訓

- **資料層 root cause 與渲染層 root cause 要分清**。直覺反應「鋸齒 = 渲染問題」，但 curl 直接看資料證明 root cause 在 OTF 字型本身（資料層）。
- **`shape-rendering` 屬性在 SVG 渲染品質有意義**。預設 `auto` 通常等於 `optimizeSpeed`，瀏覽器為了速度犧牲品質。CJK outline 字型放大時值得明示 `geometricPrecision`。
