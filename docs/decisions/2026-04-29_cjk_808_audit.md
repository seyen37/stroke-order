## 決策 1：要做 retrospective audit 而非「假設它沒問題」

**情境**：4 個後來建立的 cover-sets（4808、5021、6792、5000）都自然套用了 §六 audit 流程，但 808 是早期建的，當時 SOP 還沒成型。

**選項**：
- A. 假設它沒問題（畢竟其他 4 個都通過）
- B. 補做完整 audit（即使可能 redundant）

**決定**：B。

**考量**：
1. SOP 的精神是「對所有官方/標準資料源都跑」，不是「對新資料源才跑」
2. 補做的成本只 20 分鐘，但「**所有 cover-sets 都通過 §六**」這個 claim 才能成立
3. 若 audit 發現問題，反而是寶貴的 catch（如 4808 vs Gist 的 GB 變體）

**教訓**：SOP 寫好後對既有資產做一次 retrospective 是必要的——避免「新規矩只管新東西」的不平等執行。

## 決策 4：§六.C Taiwan-variant integrity — ✅（特殊情境說明）

**情境特殊性**：cjk_common_808 是 **中日韓三方共用**字單，本質就不是純 Taiwan-variant 資料源。但仍需檢查：

1. 有沒有夾帶明顯的 GB 變體？
2. 字的 codepoint 是否在 CJK Basic（避免 PUA / Ext B+ 錯位）？
3. simp/trad 雙欄處理是否完整？

**檢查項目**：
- [x] 確認所有字元在 CJK Basic 範圍
- [x] 對近似字（彝/彞、體/体、為/为）特別檢查
- [x] simp/trad 雙欄完整

**實證**：
- 0 個非 CJK Basic 字元（全部在 U+4E00-U+9FFF 範圍）
- ❌ 找不到 `彝` (GB18030 U+5F5D 變體)——若存在會是嚴重污染
- ❌ 找不到 `体` (GB 簡化版)
- ❌ 找不到 `为` (GB 簡化版)
- ✅ simp 254 字、trad 全 808 字，兩者並存（小學中等程度的台灣使用者也能識別）

**特殊註記**：CJK 三方共用字單的「Taiwan-variant」要求弱化為「**沒夾帶明顯 GB 變體**」。實質上 808 字大多是 G/H/T/J/K 共有字（同 codepoint），差異主要發生在 simp/trad 層級，我們已透過雙欄 schema 處理。

**結論**：C 段通過。給 808 的特殊性下，「沒夾帶 GB 變體」即為通過標準。

## 決策 5：把這個 audit 結論的 metadata 寫回 cover-set？

**情境**：JSON 沒有 `audit_status` 之類的欄位，audit 結論只散落在決策日誌。

**選項**：
- A. 在 JSON 加 `audit` 欄位記錄結果
- B. 不改 JSON，靠決策日誌 trace（git grep）
- C. 建立 `audit_log` 集中管理

**決定**：B（暫時）。

**考量**：
1. 五個 cover-sets 都通過 §六 audit，「狀態相同」不需 per-file 標註
2. JSON schema 改動會 trigger CoverSet dataclass 改動 + 測試改動，cost > benefit
3. 未來若有 cover-set audit 失敗（重要決策），再考慮 A 或 C
4. 目前的 paper trail 透過：cover-set JSON → docs/decisions/ → grep "$cover_set_name" 可建立連結

**教訓**：metadata 添加要等到實際發生「需要區分狀態」的場景才做，**不要 premature schema 升級**。
