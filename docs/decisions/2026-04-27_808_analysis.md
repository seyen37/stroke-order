## 決策 1：4 份 PDF 是否完整讀取

**情境**：使用者上傳 4 份 PUB_*.pdf（總 76 MB，34/113/352/360 頁）。讀完全部會耗大量 context，但有些可能是一次性 metadata 確認。

**選項**：
- A. 讀全部 4 份 PDF
- B. 用 pdftotext 快速掃 metadata + 第一份完整讀
- C. 完全不讀，只用使用者貼的 808 簡體字單

**決定**：B。

**考量**：
- pdfinfo + pdftotext head 能快速判斷哪份是核心
- PDF 主體內容是 chart 圖片（pdftotext 抽不出筆順表本身）
- 使用者已直接貼 808 字，主資料源確認即可
- 但 metadata 對溯源（TCS 2014 / NATF）有價值，應確認

**結果**：
- PUB_1570754349.pdf（2016, 34 頁）→ 808 表官方 Booklet（TCS 出版）
- PUB_1655945242.pdf + PUB_jp_1585037726.pdf（2019）→ TCVD 2nd ed 中日韓共用詞典（658 詞彙）
- PUB_1689037628.pdf（2023, 113 頁）→ **中日韓共用漢字集 / Workbook，含書法家手寫字帖** ← 未來 5d 的潛在 anchor reference 資料源
- 確認源頭：2014 年 NATF 第 8/9 屆會議協議產出

**教訓**：對大檔案優先 metadata 偵察，再決定深度。第二份產出意外發現「2023 workbook 含書法家手寫」是個未來重要素材線索，沒讀第二輪 PDF 就漏了。

## 決策 2：分析資料來源 — KanjiVG 部分 cache vs cjkvi-ids 完整

**情境**：要做 808 字組件分析，需要 character → component decomposition data。專案內 `data/kanjivg_cache/` 只有 55 個字（先前互動緩存）。

**選項**：
- A. 完整下載 KanjiVG（~11000 SVG, ~50MB）
- B. 改用 cjkvi-ids 的 IDS 文字檔（~2 MB, 88,939 字）
- C. 從 Unihan radical 資料推估
- D. 限縮分析範圍到 cache 內 55 字

**決定**：B（cjkvi-ids）。

**考量**：
1. **覆蓋度**：cjkvi-ids 88,937 entries 涵蓋幾乎所有 CJK 字符，KanjiVG 覆蓋 ~11000
2. **資料形式**：IDS 純文字格式比 SVG 解析快十倍
3. **下載量**：2 MB vs 50 MB，沙箱環境網路傳輸壓力小
4. **資料權威性**：cjkvi-ids 是 CHISE 衍生，學術界主流引用
5. **未來整合**：Phase B 落地時還是要 KanjiVG 的軌跡資料；現在做覆蓋率分析用 IDS 即可，不需 SVG

**權衡**：IDS 格式只給「結構描述」，沒有筆順與軌跡。但本次只做覆蓋率統計，這正好夠用。

**教訓**：分析任務先想清楚「需要什麼粒度的資料」再選資料源。SVG 級資料留給未來 Phase B 落地組件切割時用。

**情境**：使用者貼的 808 字是簡體版（来、个、亿、从、圣...）。本專案以繁體為主，分析應跑哪個版本？

**選項**：
- A. 只分析簡體（直接用使用者貼的）
- B. 分析雙版本，輸出比較
- C. 用 OpenCC 轉繁體後分析

**決定**：C（轉繁體後分析）+ 在資料集裡保留雙版本。

**考量**：

**教訓**：跨字集格式整合時，「保留原始來源 + 加上工作版本」比「直接覆蓋」安全。未來若 OpenCC 轉換有誤，可回頭比對。

## 決策 4：parser 中混入 PUA 字元的 bug

**情境**：第一次跑分析，得到「808 字 → 757 獨特組件、覆蓋率 24.7%」——數字看起來合理但偏低。檢查時發現 `ids_map['明']` 是空字串。

**Debug 過程**：
1. 隔離測試 parser，輸入單行 → 正常
2. import script 用 parser → 空字串
3. 看 script 原始檔 `xxd` → 發現 regex 字串中混入 U+E010 與 U+E01E 兩個 PUA 控制字元

**根因**：寫檔時，`r"[︀-️0-F]"` 的 regex 在 Write tool 處理過程中混進兩個不可見的 Private Use Area 字元，導致 character class 範圍變成 `[U+FE00-U+FE0F]` 加上 `[U+E010-'0']`（後者其實是反向範圍，但 Python regex 仍解析為 `'0'-U+E01E`，吞掉所有 CJK 字符）。

**選項**：
- A. 移除 regex（不處理 variation selectors）
- B. 改寫成 `\uXXXX` escape 形式
- C. 用 `chr()` 程式碼建構

**決定**：B + C 混合——`IDS_DESCRIPTORS` 改用 `chr()` build，`VARIATION_SELECTORS` 用簡化的 `[︀-️]`（只有 U+FE00-U+FE0F 這一個 range，不容易出 PUA bug）。

**修正後結果**：
- 獨特組件 757 → **194**
- 覆蓋率 24.7% → **96.1%**
- 槓桿率 ~1x → **5.16x**

**教訓**：
1. **永遠驗證 parser**：寫資料處理流程一定要先 sanity check 一個 known answer。如果先確認 `ids_map['明'] == '⿰日月'`，bug 第一秒就會被抓到。
2. **regex 字元類別用 escape 安全**：CJK 字符、Variation Selectors、PUA 範圍打字時容易混入隱形字元。優先用 `\uXXXX` 或 `chr()`。
3. **檔案處理鏈中可能有非預期字元注入**：寫長 string literal（特別是含 unicode）時，分段較安全，或用 `chr()` 程式碼。

## 決策 5：實證結果 vs VISION.md 預估的差距

**實證結果**：
- 194 獨特組件（VISION 預估 600-800）
- 96.1% 覆蓋率（VISION 預估 5-7x → ~83%）
- 槓桿率 5.16x（VISION 預估 5-7x，落在範圍內）

**問題**：要修正 VISION.md 嗎？修哪些？

**選項**：
- A. 不改 VISION，補一個附錄
- B. 改 VISION，把預估數字直接換成實證
- C. 並列兩種數字，標註理論 vs 實證

**決定**：B + C 混合——保留原本的預估表格（VISION 是策略文件，理論預估有教學價值），下面加「實證驗證」小節，寫入真實數字 + 連結到 report。

**考量**：
- 改掉預估會失去「漸進式探索」的脈絡——讀者看不到「我們怎麼從理論走到實證」
- 全部保留會讓策略文件讀起來矛盾
- 折衷：預估表保留，下面加實證，**修訂歷史明確標註修訂**

**對外承諾的修正**：
- 從「寫 600-800 字覆蓋 3500」→「**寫 808 字 → 可組合 4171 字**」（實證有依據）
- Phase A 任務清單加「內建 808 表為預設覆蓋集選項」（已落地的具體決定）

**教訓**：策略文件的數字應該分層：理論預估（給直覺）+ 實證 baseline（給承諾）。對外溝通用實證，內部討論用理論的張力。

## 決策 6：分析腳本是否要進 repo

**情境**：`build_808_dataset.py` 與 `analyze_808_components.py` 都是一次性分析腳本，未來可能不需要再跑。要進 repo 還是只放工作目錄？

**選項**：
- A. 全部進 repo（給未來人重現分析）
- B. 只 commit 結果（json + md），腳本扔掉
- C. 進 repo 但放 `scripts/` 資料夾標明是 one-off

**決定**：C。

**考量**：
1. 重現性：別人質疑數字時，能跑出同樣結果是負責任的科學態度
2. 程式碼是文件——`analyze_808_components.py` 本身解釋「我們怎麼定義覆蓋率」「為什麼 96.1%」，比口頭說明清楚
3. 體積極小（兩個檔加起來 < 15KB）
4. `scripts/` 資料夾 imports 不影響主套件 build
5. 未來若使用者選別的字集（教育部 4808、5000 通用），可直接 fork 這個腳本改

**教訓**：一次性分析腳本進 repo 的成本低、價值高。寧願留著看似冗餘的程式碼，也不要事後找不回某個結論的計算過程。

## 決策 7：IDS 資料是否進 repo

**情境**：cjkvi-ids 的 `ids.txt` 是 2 MB，分析結果依賴它。

**選項**：
- A. ids.txt 進 repo（保證離線可重現）
- B. 不進 repo，腳本提示使用者下載
- C. 抽取 808 字 + 3500 字相關的 subset 進 repo

**決定**：B。

**考量**：
1. cjkvi-ids 是 BSD 授權的開源資料，不會消失
2. 2 MB 對 repo 不致命，但增加 clone 大小無實質好處
3. 分析腳本第一行就 print 下載指令，學習成本極低
4. 若未來要分析別的字集，subset（C 方案）會限制範圍

**實作**：`analyze_808_components.py` 開頭 docstring 寫明下載指令；找不到 `ids.txt` 時印錯誤訊息加下載命令，使用者一行 curl 解。

**教訓**：第三方開源資料 + 公開穩定 URL 的情況，不需要進 repo。**進 repo 的標準是「資料若消失我們會失去什麼」**，cjkvi-ids 失消失機率極低。

## 反思

**最值得記下的**：
- 「第一次結果不對勁就一定要 sanity check」這個工程習慣救了一次。如果沒去檢查 `ids_map['明']`，VISION.md 會被錯誤的 24.7% 數字污染
- 從「使用者貼了一份字單」到「實證驗證 VISION 假設並改寫文件」，整個鏈路 ~90 分鐘——這是 AI + 結構化資料 的合作能交付的價值
- TCS 2014 三方學者的協議結果跟組件演算法的最佳解方向一致，這個觀察值得寫進對外宣傳——「**手工選的高頻共用集合 ≈ 演算法最小覆蓋集**」是個有故事性的洞察

**反面教訓**：
- regex unicode range 永遠用 escape 形式寫，避免 PUA 注入
- 多步驟流程要 sanity check 每一步輸出，不要一路跑到底再驗證
