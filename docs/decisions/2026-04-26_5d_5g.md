## 決策 1：寫字機器人軌跡格式

**觸發**：5d 規劃 PSD 時，要決定要存什麼欄位

**選項**：

| 編號 | 方案 | 優點 | 缺點 |
|---|---|---|---|
| A | 只存座標 (x, y) | 最小體積 | 機器人模仿不出真人手寫節奏、無筆鋒 |
| B | (x, y, t_ms) 加時間戳 | 可重現節奏 | 缺筆壓 |
| C | (x, y, t_ms, pressure) | 含筆壓變化 | 觸控筆才有實值 |
| D | 加 tiltX / tiltY 觸控筆角度 | 最完整 | 大多數裝置沒實值 |

**考慮的因素**：
- 寫字機器人最終目標是**模仿真人**——光座標等速描繪會像「打字機」
- 觸控筆有 pressure + tilt，滑鼠/手指沒有 → fallback `0.5` pressure / `0` tilt
- 體積：6 維 vs 2 維每個取樣點多 4 個 number ≈ 多 60% 體積，但 60Hz 取樣下 1 字才 ~3KB → 不影響

**選擇**：☑ D（全收）

**理由**：
> 體積差別小（KB 級），但對寫字機器人模擬真人手寫**質的差距**。沒有的欄位 fallback 預設值即可，不影響無 pressure/tilt 裝置使用。座標統一 EM 2048（與既有 stroke-order 引擎對齊），方便未來餵 plotter pipeline。

**後續驗證**：✅ 5d MVP 完成、Plotter SVG 匯出含 `data-avg-pressure` 屬性、tests 覆蓋

## 決策 2：JSZip vendor vs 自寫 ZIP encoder

**觸發**：5d-10 多檔 plotter SVG 下載

**選項**：

| 編號 | 方案 | 優點 | 缺點 |
|---|---|---|---|
| A | 拉 JSZip vendor (~90KB) | 業界標準、功能完整、可解壓 | 額外 dependency / 維護 |
| B | 自寫 STORE-mode encoder (~120 行) | 零依賴、自包含 | 只能產生不能解壓、手刻可能有 edge bug |
| C | 逐字下載不打包 | 最簡單 | 50 字下載很煩 |

**考慮的因素**：
- 我們**只需要產生不需要解析** zip → JSZip 大半功能用不到
- ZIP STORE 模式（不壓縮）格式很簡單，120 行解決
- 90 KB 一次性 vendor download vs 120 行原生程式碼
- senior preference: 「touch minimal scope」、零依賴

**選擇**：☑ B（自寫 minimal encoder）

**理由**：
> JSZip 的功能 80% 我們用不到，部署多一個 vendor 檔的維護負擔不值得。120 行純 JS（含 CRC32 表）可以通過 Python `zipfile` round-trip 驗證（含 CJK 檔名 UTF-8 旗標 0x0800）。事後事實證明選對：完全沒踩坑。

**後續驗證**：✅ Python `zipfile` round-trip 通過，CRC32 全綠，CJK 檔名正常

**事後備註**：用戶原本選 B (JSZip)，我跟他解釋自寫 encoder 工程量小於拉 vendor，他同意改為自寫。最終選的是「自寫」而非他原本投票的「JSZip」。

## 決策 3：5d 警語措辭精確性

**觸發**：5cb 隸書/篆書 priority hint 顯示後，使用者反饋「您寫的每一筆都會幫助累積真實軌跡資料庫」這話不準確

**選項**：

| 編號 | 方案 | 優點 | 缺點 |
|---|---|---|---|
| A | 維持原文（鼓勵性質）| 動機強 | 誤導：實際只存本機 |
| B | 精準描述「先存本機 → 日後可選擇匯出/提交」 | 誠實 | 鼓勵性減弱 |

**考慮的因素**：
- 工程倫理：不能誤導使用者以為「我寫的每筆都自動上傳」
- 5g 還沒做，使用者寫的東西**真的**只存本機
- 鼓勵收集動機可以靠別的設計補（5g 之後加點讚機制）

**選擇**：☑ B

**理由**：
> 工程倫理 > UX 動機。誤導訊息會破壞長期信任。**B 改寫成「您寫的軌跡會先存在本機資料庫；日後可選擇匯出 JSON 或提交至公眾資料庫（5e 階段開放），協助寫字機器人取得真實軌跡」**——既誠實、又預埋了未來 5g 的鉤子。

**後續驗證**：✅ 5cb fix 已 ship，5g 完成後使用者真的可以「選擇提交」

**學到**：工程精準性 > UX 包裝。每個訊息都要對得起實際行為。

## 決策 4：5g 使用者識別方式

**觸發**：5g 規劃需要鎖定 auth 方式

**選項**：

| 編號 | 方案 | 優點 | 缺點 |
|---|---|---|---|
| A | email magic-link | 簡單、跟未來 5e SMTP 共用 infra、點讚可信度高 | 需 SMTP 帳號 |
| B | 自定 handle 不驗證 + IP 冷卻 | 完全不需 SMTP | 點讚可信度低、共享 IP 環境（咖啡店）會錯傷 |
| C | 純匿名 + browser fingerprint | 最低門檻 | 換瀏覽器就重置、無檔案歸屬 |
| D | OAuth (Google/GitHub) | 信任度最高 | 太重、依賴第三方 |

**考慮的因素**：
- 5e 規劃就要做 SMTP（寄 email 提交副本給使用者）→ 跟 magic-link 共用
- 點讚機制要避免一人灌水 → 需要可信識別
- 部署門檻：A 需要 SMTP 設定，B/C 完全不需要、D 要申請 OAuth app
- Dev mode 可以擋掉 A 的部署門檻（本機測試不需 SMTP）

**選擇**：☑ A（email magic-link）

**理由**：
> 跟 5e 規劃 infra 共用、點讚信任度足夠、有 dev mode 解決部署門檻。OAuth 太重（多了 callback URL 設定 + Google/GitHub app 申請）；自訂 handle 失去點讚意義（使用者一人灌水）。

**後續驗證**：✅ dev mode 在 sandbox / CI 都跑通；正式部署只需 Gmail App Password 5 分鐘設定

## 決策 5：5g 點讚機制

**觸發**：5g 規劃 Q3——點讚怎麼分配

**選項**：

| 編號 | 方案 | 優點 | 缺點 |
|---|---|---|---|
| A | 每個使用者一輩子 X 點 | 限量 → 慎重 | 上傳變懲罰（用一個少一個） |
| B | 每次上傳得 2 點（自己 1 + 給他人或棄權 1） | 鼓勵上傳 | 灌水可能性 |
| C | 完全不限量 | 自由 | 互讚刷分 |

**使用者明確的偏好**：B 並加重複偵測（重複內容只給自己 1 點，沒第 2 點可給他人）

**選擇**：☑ B + 重複偵測

**理由（使用者的）**：
> 鼓勵上傳的同時防灌水——重複內容只能自己加分，沒法幫別人加分。這個設計把「灌水」變成「自爽不影響別人排名」。

**實作細節**：
- 上傳時 SHA-256 比對既有資料庫
- 完全新內容 → 2 點
- ≥50% 重複 → 1 點（只能給自己）
- 完全重複（已存在）→ 拒絕上傳
- 自己給自己 1 點**自動加上**（不需手動按）
- 永久累積總分 + 週/月/年排行**分開計算**

**後續驗證**：⏸ 待 5h 實作（5g MVP 還沒做點讚）

## 決策 6：5g 階段切分

**觸發**：5g 範圍太大（auth + DB + uploads + 點讚 + 排行 + 抄經快取），不能一次做完

**選項**：

| 編號 | 方案 | 工程量 | 部署門檻 |
|---|---|---|---|
| A | 5g 一次做完 Block A + B + 點讚 | 太大 | 風險高 |
| B | 5g (Block A-1: 基礎) → 5h (點讚) → 5i (Block B 抄經快取) | 切三段、逐次見效 | 中 |
| C | 5g (Block B 抄經快取先) → 5h (Block A 上傳) → 5i (點讚) | 先解負載問題 | 高（需 SMTP 才上線 A） |

**考慮的因素**：
- Block A（PSD 上傳）是使用者**核心願景**——焦點是鼓勵收集
- Block B（抄經快取）是**附帶效率優化**——nice-to-have
- 點讚機制需要使用者識別 → 必須在 Block A 之後
- 跟原規劃的 5e (SMTP) / 5f (PSD 注入) 衝突——需要重排

**選擇**：☑ B（Block A → 點讚 → Block B）

**理由**：
> 核心願景（鼓勵收集 PSD）優先。Block A-1 給「上傳 + 列表」基本骨架先 ship，5h 加點讚遊戲化，5i 才做負載優化。原規劃的 5e/5f 重排：5e 的 SMTP 部分提前到 5g 用了；OCR 跟 PSD 注入延後到累積實際資料看效果再決定。

**後續驗證**：✅ 5g MVP（Block A-1）已上線

**事後備註**：使用者沒爭議直接接受。後來執行時也驗證了：Batch 1 (DB+SMTP+auth) → Batch 2 (service+endpoints) → Batch 3 (前端) → Batch 4 (bump+docs) 切分清晰。

## 決策 7：5g 後端依賴（PyJWT / aiosmtplib / ORM）

**觸發**：5g auth + DB + SMTP 都有業界標準依賴可選

**選項**：

| 子系統 | 業界標準 | stdlib 替代 |
|---|---|---|
| Token 簽章 | PyJWT | hmac + base64 自簽 (~25 行) |
| SMTP | aiosmtplib | smtplib + asyncio.to_thread |
| ORM | SQLAlchemy / SQLModel | sqlite3 raw |

**考慮的因素**：
- 既有專案**零 web 框架以外的執行期依賴**（除 fastapi/uvicorn/cairosvg/Pillow）
- 加新依賴會增加部署門檻、維護負擔、安全更新責任
- HMAC token 規格穩定（25 年）
- SQLite schema 簡單（4 張表），ORM 收益不高
- senior preference: 自包含、touch minimal scope

**選擇**：全部 ☑ 用 stdlib

**理由**：
> 所有功能 stdlib 100% 覆蓋且程式碼量都很小。PyJWT 大半功能（多 algorithm、JWK、RFC 各種選項）用不到；aiosmtplib 只是把 smtplib 包成 async（我們 `asyncio.to_thread` 解決）；ORM 對 4 張表 + 簡單 query 過度設計。

**後續驗證**：✅ 全部 19+24=43 條測試通過、forged-token 防偽機制有效（HMAC 過得去但 DB 沒登記就拒絕）、TestClient 通過 SMTP dev mode + magic-link end-to-end

## 決策 8：5g cookie 政策（httpOnly / SameSite / Secure）

**觸發**：5g session cookie 設計

**選項**：

| 屬性 | 選項 | 我們選 |
|---|---|---|
| httpOnly | ✅ / ❌ | ✅ |
| SameSite | Strict / Lax / None | **Lax** |
| Secure | True / False | False（dev 友善），生產靠反向代理 |
| max-age | session / 持久 | 30 days |

**考慮的因素**：
- Strict 會讓「使用者點 email 連結 → consume → set cookie + redirect」流程被擋（外部 site 帶來的 GET 請求不送 cookie）
- Lax 允許 top-level navigation 帶 cookie，正好支援 magic-link 流程
- httpOnly 防 XSS 偷 cookie
- Secure=True 在 HTTP dev 環境會直接不下發 cookie → 設 False，文件提醒生產用 HTTPS + 反向代理 ProxyHeaders

**選擇**：☑ httpOnly + SameSite=Lax + Secure=False（生產靠 reverse proxy）+ 30 day

**理由**：
> 在「dev 友善」跟「生產安全」之間找平衡。dev mode 不必設 HTTPS 也能登入測試；生產用 Caddy/nginx + `X-Forwarded-Proto: https`。SameSite=Lax 讓 magic-link 流程能無縫運作。

**後續驗證**：✅ 24 條 API 測試通過 + 部署文件詳細說明 HTTPS 反向代理設定

## 學到的規則 / pattern（適用未來）

### SVG 渲染相關

1. **任何 `<g transform="scale()">` 內的 stroke**：必須用 `vector-effect="non-scaling-stroke"` 或預先反向縮放 stroke-width，否則細線會被縮到亞像素消失
2. **outline 渲染**：先檢查座標系是否 EM 全填——多種字型源（kaishu / lishu / seal_script）的 EM bbox 中心可能不在 (1024, 1024) → 要 bbox-center 對齊
3. **描紅紙視覺強度**：~12% 灰（`#cccccc × 0.55 ≈ #e1`）符合書法描紅紙觀感

### 測試 / 驗證相關

4. **測試強度層次**：「DOM 屬性正確」≠「視覺看得見」。SVG transform + stroke 類修改必須加 raster 像素層級驗證
5. **TestClient cookie jar 是 per-client 持久**：多使用者場景**用 per-user TestClient**，不要在單一 client 上交替切換 `cookies=` kwarg
6. **FastAPI Path 命名衝突**：`Path` 與 `pathlib.Path` 衝突 → `from fastapi import Path as ApiPath`

### 架構 / 取捨相關

7. **vendor vs 自寫**：對「只需要產生不需要解析」的功能，自寫常贏在零依賴 + 部署乾淨
8. **dev mode 的價值**：讓需要外部服務（SMTP / OAuth）的功能在沒設定的環境也能跑——大幅降低試用門檻
9. **獨立 SPA 子頁路線**：複雜模組獨立 SPA + ES modules + 純 fetch /api/*，不和主頁共享狀態，維護單純

### 流程 / 規劃相關

10. **大功能先寫詳細 spec、確認後才動代碼**：避免做到一半發現方向錯
11. **Batch 切分按使用者感受**：每批結束都有可看到的進展，使用者隨時可踩煞車
12. **訊息文案精確性 > UX 包裝**：每個訊息都要對得起實際行為，誤導會破壞長期信任
