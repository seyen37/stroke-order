### 關鍵 root cause 發現

**12b-5 找到 11e/11g 連鎖暴露的舊 bug**：`inner_w = max(width_mm - 2*inset, char_size_mm)` 過去用 EM-based scale 字身有自然 padding，bug 不顯眼；11g 改 bbox-based scale 字撐滿 cell 後，char_size_mm 污染 inner 的副作用就立刻造成「user 改字大小章面 inset 也跟著變」的怪現象。

→ 寫進 decision log 的「教訓」：**底層改動會暴露上層舊 bug**（11g 改 scale → 12b 才暴露 inner_w 計算 bug）。

### 12b-1 業界 8 級尺寸 quick-pick

- preset 切換時也同步更新 quick-pick

### 12b-2 小尺寸警示

`width ≤ 12mm + 字數 ≥ 3` → 警示「⚠ 1.2cm 以下尺寸 + 3 字以上時筆劃可能相連」。
`width ≤ 15mm + 字數 ≥ 4` → 升級警示「建議放大尺寸或減少字數」。

### 12b-3 字數推薦尺寸

```
1 字 → 1.0–1.5 cm
2 字 → 1.0–1.8 cm
3 字 → 1.2–2.1 cm
4 字 → 1.5–2.4 cm
5+ 字 → 2.4–3.0 cm（公司章常見）
```

### 12b-4 PDF 下載

後端：`POST /api/stamp` 加 `format=pdf` 支援，cairosvg svg2pdf **直出**（單頁，不走 sutra 的 SVG→PNG→Pillow 路線）。
前端：「⤓ PDF」按鈕加在 SVG 跟 G-code 之間。
測試：原 invalid-format 測試從 pdf 改成 docx，新增 PDF format 正功能測試。

### 12b-5 char_size_mm bug fix

兩個 bug：
1. **inner_w/h 計算**：`max(_, char_size_mm)` → 改成 `max(_, 1.0)` 純安全下限
2. **square_name n=1/2/4 用 char_size_mm 當字身**：改成 cell-based size，char_size_mm 變成 cap

UI 同步：「字大小」label → **「字大小上限」**，min 改 0（允許「完全撐滿 cell」），tooltip 說明語意。

### 12b-6 border_padding 預設值

7 處 default 同步：`server.py × 3` + `stamp.py × 4` + `index.html × 1` 全部 `2.0 → 0.8`。

業界小章 4-7 分 inset ≈ 0.5-1.0 mm；過去 2.0 是公司章規格用在小章上不合適。

### 12b-7 cell 內留 8% padding

`square_name` n=1/2/4 cell_size 乘 `CELL_FILL_RATIO = 0.92`，防 4 字 2×2 字邊互相碰邊。3 字 1+2 layout 已有 `inner_h * 0.92` 比例不需此處再縮。

## 八、給未來自己的話

1. **「user 提供高品質參考圖」是推進效率的最大乘數**：
   8 張範例圖讓我能在 30 分鐘內整理出業界共識，並做出 5+ 個對齊決策。如果只靠口頭描述「想做傳統印章」，光是釐清標準就要花一個小時以上。下次任何「對齊外部標準」的工作，**先請 user 提供 3-10 個範例圖** 永遠是 ROI 最高的開場。

2. **底層改動會暴露上層舊 bug**：
   11g（11 號階段 g 子任務）改 bbox-based scale，本身只是修「字沒撐滿 cell」的視覺問題；但因為 bbox-scale 比 EM-scale 更「貼合」，把 inner_w 計算的舊 bug（max 副作用）放大到使用者可感知。**重要 base infra 改動後，要主動 audit 周邊邏輯**，不能假設「只有預期範圍會被影響」。可考慮寫進 §8.x 候選原則。

3. **default 值散布在多處是反模式**：
   `border_padding_mm=2.0` 散布在 7 個地方（3 server + 4 stamp.py + 1 index.html）。這次 7 處同步改是手動 grep + sed，但任何一處遺漏就會造成 inconsistency。**理想做法**：constants 集中在一處（如 `stamp.py` 頂端 `DEFAULTS = {...}`），其他地方 import。本次沒重構是因為改動範圍可控（grep 找得齊），但這個 debt 值得列入 backlog。

4. **「user 主觀感受」+「業界客觀標準」雙管齊下**：
   本日同時用「user 上傳的截圖」（主觀）+「8 張業界範例圖」（客觀）兩個資料源做決策。雙管齊下的好處：客觀標準避免 over-fit user 個人偏好；user 主觀避免 over-engineer 不必要的細節。**任何 UX 設計工作都建議用這個雙資料源模式**。

## 十一、12c 給未來自己的話

5. **演算法工作的 SOP：prototype-first**：
   12c 整個過程主分支 1 commit 就完成了 5 子任務 + 5 測試。**關鍵是先在 `scripts/` 寫 prototype 驗證演算法**（scanline + even-odd），確認效能 + 視覺 + G-code 規模都 OK 之後才動主線。如果直接動主線寫演算法 + 接 API，會變成「邊改邊試」式 churning，主分支多輪 commit + 多次驗證紅。**值得寫進 PROJECT_PLAYBOOK §8.11**：「未驗證可行性的演算法工作，動主線之前先寫 prototype」。

6. **演算法 module 獨立、不污染既有 file**：
   `exporters/engrave.py` 是新檔，stamp.py / patch.py 都 import 用。stamp.py 已 1100+ 行，再塞演算法進去 god file 會更難維護。同時 module 不依賴 stamp/sutra/patch 業務概念，**保留跨用途複用的彈性**（patch 鏤空、sutra watermark 都能用）。跟 12b §8.10 (default single source of truth) 屬於同一個原則 cluster：「物件職責邊界要清楚」。

7. **「向後相容性」當作預設假設**：
   `engrave_mode` 預設 `"concave"`，所有既有 caller 不傳此參數就行為不變。這讓 Phase 11/12b 累積的 47 個 stamp 測試 100% 維持，無需改一個字。**新功能加入既有 API 時，default 必須對應「既有行為」**——不是「新功能的合理預設」。
