# stroke-order：核心願景

> 最後更新：2026-04-27 · 對應 v0.13.0
>
> 本文件定義 stroke-order 專案的長期戰略方向。所有後續開發決策（功能優先級、資料結構演進、API 設計）都應對照這份願景檢查是否一致。

---

## 一句話定位

**世界上第一個「真實手寫軌跡 × 組件化字型 × 個人風格生成」三者交集的開源系統。**

不是筆順動畫教學工具。不是字帖 generator。不是書法 GAN model。是把寫字機器人 / 個人字型 / 漢字結構研究 三個領域共同需要、但目前都缺的那塊基礎建設補上。

---

## 二、核心洞察：為什麼現有系統都不對

漢字筆跡 / 字型 / 寫字機器人領域，現存方案都在解局部問題：

| 領域 | 現存代表 | 解決什麼 | 沒解決什麼 |
|---|---|---|---|
| 筆順教學 | 教育部、hanzi-writer | 給人類看的順序動畫 | 真實手寫節奏、寫字機器人軌跡 |
| 開源軌跡資料 | KanjiVG、Make Me a Hanzi | 字級的合成 polyline | 真實人手節奏、組件變形規律 |
| 字型組件化 | KAGE 引擎、GlyphWiki、Wenlin CDL | 輪廓字型的組件組合 | 軌跡層級、個人風格、書寫節奏 |
| 書法字型生成 | zi2zi、CalliGAN、AEGG | 字形 raster 風格遷移 | 向量軌跡、可動畫、寫字機器人能用 |
| 線上手寫資料 | CASIA-OLHWDB | 辨識用的整字資料 | 組件結構標註、生成導向 |

每個方向各做各的。**沒有任何一個系統把「真實人手軌跡」「組件結構」「個人風格」三者同時納入**。這就是 stroke-order 要佔的位置。

## 三、組件化哲學

朱邦復 1976 起的明珠字型／倉頡系統，提出了一個技術上正確、但商業上失敗的願景：**用組件 + 變形規則生成全字型**。日本 GlyphWiki / KAGE 引擎用相同哲學做出開源輪廓字型基礎設施。但兩者都停留在 **outline 層**，沒人把它推到 **trajectory 層 + real handwriting 層**。

stroke-order 的核心主張：

> **「同一個使用者書寫同一個組件時，軌跡有可重複的個人風格指紋。」**
>
> **「常用 3500 字可分解成 ~600-800 個獨特組件 × 6-8 種位置變體。」**
>
> **「使用者寫 600-1000 個精選字，可以覆蓋整個常用組件空間，剩下的字交給組合演算法生成。」**

這三句話加起來等於一個結論：**個人字型不必寫滿 3500 字，可以寫 1/3-1/5 字數覆蓋組件空間，其餘自動推導。**

## 四、覆蓋率數學

為什麼這條路在數量上 work：

| 字集規模 | 字數 | 獨特組件數（CDL/KAGE 級分解）| 真寫:組合比 |
|---|---|---|---|
| 國中小常用 | ~3,500 | ~500-700 | 1:5-7 |
| 教育部標準字 | ~4,808 | ~600-800 | 1:6-8 |
| 通用漢字 | ~7,000 | ~700-900 | 1:8-10 |
| Unicode 基本平面 | ~27,000 | ~1,000-1,300 | 1:20-27 |

**愈大的字集，組合策略的槓桿率愈高**——寫 1000 字覆蓋 27000 字，這就是組件化路線的本質回報。

### 實證驗證：中日韓共用 808 漢字（TCS 2014）

2014 年三方協議 `中日韓共同常用八百汉字表` 的 808 字提供了**現成的、有國際背書的最小覆蓋集候選**。我們用 CHISE/cjkvi-ids 的 IDS 分解資料跑了實證分析（`scripts/analyze_808_components.py`），結果：

| 指標 | 實證數值 |
|---|---|
| 808 字分解出的獨特葉組件 | **194** |
| 平均每字葉組件數 | 3.77 |
| 完全覆蓋常用 3500 字的比例 | **96.1%**（3363 字） |
| 部分覆蓋 | 3.1%（108 字） |
| 完全未覆蓋 | 0.8%（29 字） |
| **真寫:組合槓桿率** | **5.16x** |

完整報告：[`docs/analysis/808_coverage_report.md`](analysis/808_coverage_report.md)。

**關鍵啟示**：
1. 實證的「**194 獨特組件**」遠少於原本估的 600-800——常用 3500 字的組件空間實際上比想像中更小。組件化路線的可行性比 VISION 初稿還樂觀。
2. 808 字的選字邏輯雖然不是「組件覆蓋最大化」（是高頻共用），但**意外地接近實用上的最小覆蓋集**——三國學者協商出的字單，跟組件演算法給出的解殊途同歸。
3. 「寫 808 字 → 可組合 ~4171 字（808 自身 + 額外 3363 推導）」是個有實證支持的承諾，可以對使用者明確說。

選哪些字當「最小覆蓋集」是 NP-hard 的 Set Cover Problem，但貪婪演算法在實務上效果非常好：

```
def greedy_cover_set(target_chars, kanji_vg_data):
    uncovered_components = {(c, pos) for char in target_chars
                            for c, pos in components_of(char)}
    cover_set = []
    while uncovered_components:
        best_char = argmax(target_chars,
                          key=lambda c: |components_of(c) ∩ uncovered_components|)
        cover_set.append(best_char)
        uncovered_components -= components_of(best_char)
    return cover_set
```

可以做成「**動態任務指派**」：使用者寫一個字後，系統即時算下個 information gain 最大的字推薦，使用者覺得每寫一個字就在拓展「能寫的字」邊界。

## 五、三層資料架構

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: 標註層（靜態，公共）                                │
│                                                             │
│   來源：KanjiVG SVG kvg:element 標註                         │
│        + Unicode IDS 結構描述                                │
│        + CHISE 漢字構造資料庫（補充）                         │
│                                                             │
│   產出：「字 ↔ 組件樹 ↔ 位置」雙向索引                        │
│        + 每組件在不同位置的「業界標準變形樣貌」統計            │
│                                                             │
│   時序：一次性離線處理，幾分鐘到幾小時                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: 個人層（動態，使用者私有）                          │
│                                                             │
│   來源：5d PSD —— 使用者的真實手寫紀錄                       │
│         (整字級 polyline + 時間戳 + 壓力 + tilt)             │
│                                                             │
│   產出：用 Layer 1 的標註自動切割成「組件級樣本」              │
│        + 每組件樣本包含使用者個人風格指紋                      │
│        + 即時更新「使用者組件覆蓋率」                          │
│                                                             │
│   時序：使用者寫一字 → 毫秒級切割 + 儲存                      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: 組合層（推論，產出）                                │
│                                                             │
│   輸入：使用者請求一個沒寫過的字 X                            │
│                                                             │
│   流程：                                                     │
│     1. 查 Layer 1 拿 X 的 IDS 結構樹                         │
│     2. 對每個 leaf 組件，查 Layer 2 是否有使用者實例          │
│        ├─ 有 → 用使用者本人軌跡                              │
│        └─ 無 → fallback 到 Layer 1 標準形                    │
│     3. 套用 Layer 1 的「位置變形規則」調整每個組件             │
│     4. 黏合 + 重心調整 → 完整字軌跡                           │
│                                                             │
│   產出：一個帶「信心度」的合成軌跡，可寫字機器人直接使用        │
│                                                             │
│   時序：百毫秒級即時推論                                      │
└─────────────────────────────────────────────────────────────┘
```

關鍵洞察：**個人層（Layer 2）愈累積愈多 → 組合層（Layer 3）的產出愈像本人**。一個使用者寫了 50 個字，可能 60% 的合成字還在用 Layer 1 的標準形；寫到 1000 字時，95% 的合成字都是用本人組件素材組合而成。**這是一個有實用價值的進度曲線**。

## 六、跟既有系統的差異化定位

| 比較對象 | 我們相同處 | 我們獨特處 |
|---|---|---|
| KAGE / GlyphWiki | 組件化合成思想 | **軌跡層級**（KAGE 是輪廓） |
| Make Me a Hanzi | 開源軌跡資料 | **使用者個人化** + 組合生成 |

| zi2zi / CalliGAN | 風格化輸出 | **向量軌跡** + 寫字機器人能用 |
| 商業手寫字型生成（Glyphs.app 之類） | 個人字型概念 | **開源** + 寫 1/5 字覆蓋全字 |
| CASIA 手寫資料集 | 真實手寫資料 | **生成導向** + 結構標註 + 實時應用 |

## 七、工程實作路線

從目前 v0.13.0 出發，組件化路線分四個階段：

### Phase A：組件覆蓋分析器（1-2 週）

- 整合 KanjiVG `kvg:element` + cjkvi-ids IDS 解析器到 IR
- 建「字 ↔ 組件清單」雙向索引（cache 到 SQLite）
- **內建「中日韓共用 808 漢字」作為預設覆蓋集**（已實證 96.1% 覆蓋率）
- 實作貪婪 set cover 演算法（讓使用者選自己的目標字集）
- 5d UI 加「下一字推薦」+ 組件覆蓋率儀表板
- 5d UI 加覆蓋集選項：(1) 中日韓 808、(2) 自動最小覆蓋集、(3) 教育部 4808、(4) 自訂

**驗收標準**：使用者打開 5d，看到「你已寫 N 字，覆蓋 M 個組件（已可組合 ~K 字），建議下一字寫 X（+k 個新組件）」。

### Phase B：組件級 PSD 切割（2-3 週）

- 寫「KanjiVG 對齊器」：把使用者整字軌跡比對 KanjiVG 標準軌跡，切成組件級樣本
- IndexedDB schema 加 `ComponentSample` 物件
- 每樣本帶 metadata：所屬字、位置、節奏特徵、原始壓力曲線
- 5d UI 加「我的組件庫」頁面

**驗收標準**：使用者寫了 100 個字 → 自動產生 ~250 個組件樣本，可在「組件庫」按組件名瀏覽。

### Phase C：規則式組合引擎（4-6 週）

- 實作 IDS parser（解析 ⿰⿱⿴ 結構描述符）
- 實作組件位置變形規則庫（左旁壓窄、上部矮化、包圍縮放等）
- 實作軌跡黏合演算法（連接點對齊、重心調整）
- 加 `/api/compose/{char}` endpoint：給字、產合成軌跡

**驗收標準**：使用者寫了 600 字 cover set → 系統能合成「鬱、龘、麤」這類沒寫過的字，視覺上「七成像本人」。

### Phase D：神經組合模型（半年起跳，研究級）

- 用 Phase B 累積的真實組合樣本訓練 Transformer
- 輸入：組件軌跡序列 + 目標位置 → 輸出：合體後軌跡
- 比 Phase C 規則式自然
- 需至少 5000 字訓練樣本

**驗收標準**：盲測使用者本人 vs 模型輸出，識別率 < 70% 即達標。

## 八、戰略價值

走組件化路線，stroke-order 的市場定位質變：

| 維度 | 整字級系統（目前） | 組件化系統（願景） |
|---|---|---|
| 使用者投入 | 寫 3500 字才有完整字庫 | 寫 600 字覆蓋常用字 |
| 價值主張 | 教學工具 + 寫字機器人玩具 | 個人字型生成器 + 寫字機器人字源 |
| 商業空間 | 小眾 hobbyist | 字型授權 + B2B（書法家個人字型） |
| 學術定位 | 工程整合 | 開創性研究（首個三維交集系統） |

更重要的是：**這條路對寫字機器人領域是真正未開發的處女地**。KAGE 在輪廓字型已經做了 20 年，沒人接力到軌跡 + 真實手寫層。stroke-order 走通的話，會在這個 niche 變成 reference implementation。

## 九、誠實列出未解問題

戰略文件不該只講樂觀面。以下是**真正的技術不確定性**，未來開發要邊走邊解：

1. **位置變形規則的完整性**：6-8 種位置（左/右/上/下/包圍/獨立...）是否足夠？「鬱」「龘」這類超複雜字可能需要更細位置劃分。
2. **組件邊界的歧義**：「必」是 心+丿 還是原子字？「我」可拆嗎？這類字學界本身就有爭議。
3. **個人風格的可遷移性**：使用者左旁「木」的風格，是否真的能套用到底部「木」？需累積資料驗證。
4. **黏合美學的演算法化**：「連勢、呼應、重心」這些書法美學概念，規則式引擎能逼近多遠？多遠之後必須上 ML？
5. **冷啟動的閾值**：使用者要寫多少字，組合輸出才從「七成像本人」跳到「九成像本人」？需實證。
6. **異體字 / 古字 / 合文**：「龍」的繁簡 / 異體變體怎麼處理？篆書、隸書是否同一份組件庫可生成？

這些問題不解，路線仍可推進——但落地時的品質取決於這些問題的解答深度。

## 十、為什麼這個路線值得堅持

技術上：把朱邦復 1976 提出但未落地的願景，加上 50 年後才有的真實手寫資料 + ML 工具，重新做一次。這在哲學的優雅和工程的可行性之間找到了甜蜜點。

人文上：個人化手寫字型在數位時代有獨立價值——讓書法家、教師、藝術工作者，能用 1/5 的工作量留下自己的字跡，未來還能寫他從沒寫過的字。**這是把書寫這件事的個人性 / 永續性 / 可機器化三者同時兼顧的方案**。

商業上：寫字機器人市場、個人字型生成市場、漢字研究學術市場——三個市場交集處目前無人佔據。

策略上：stroke-order 已經有 1057 條測試 / 9 個 Web UI 模式 / 開源 license / 線上 demo / 完整文件——基礎設施齊全。組件化路線是「下一個量級」的功能擴展，建立在現有地基上不需打掉重練。
